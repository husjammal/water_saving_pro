1 Introduction
This document provides a comprehensive overview of the Water Monitor Flut- ter application and its companion firmware (code.py) for submission to test- ing. It includes detailed explanations of the application’s functionality, archi- tecture, setup instructions, file structures, and testing steps. Screenshots are at specific points to enhance clarity.

2 Application Overview
2.1 Purpose
The Water Monitor application enables users to:
• Connect to a water monitoring device via Bluetooth Low Energy (BLE).
• Retrieve and store water usage data (timestamp, flow rate, battery voltage, tap-on duration).
• Visualize live and historical data through charts.
• Monitor SD card status on the device.
• Synchronize the device’s time and manage CSV data storage.
The firmware (code.py) manages device-side operations, including data log- ging, SD card interactions, and BLE communication.

2.2 Key Features
1. Bluetooth Connectivity: Scan and connect to BLE devices with UART ser- vice UUID 6e400001-b5a3-f393-e0a9-e50e24dcca9e.
2. Data Retrieval: Fetch incremental data or all data and store in a CSV file.
3. Data Visualization: Live and historical charts for flow rate, battery volt- age, and tap-on duration.
4. SD Card Monitoring: Check SD card presence, file existence, size, and last modified time.
5. Time Synchronization: Sync device time with phone time adjusted for UTC offset.
6. Data Management: Clean, clear, and share CSV data.

3 System Architecture
3.1 Flutter Application Structure
The application follows a Model-View-Controller (MVC) pattern with Provider for state management:
• Models: ConnectionStateModel and DataStateModel manage Bluetooth and data.
• Views: Screens render the UI.
• Controller: Logic in screens and models handles interactions.
Key dependencies include flutter_blue_plus, fl_chart, and provider.

3.2 Firmware (code.py) Overview
The firmware runs on a microcontroller with CircuitPython, managing BLE com- munication, SD card operations, and command processing. Supported commands include GET_SD_STATUS, GET_DATA, SYNC, and PING.

6 Detailed Component Description
6.1 Flutter Screens
• SplashScreen: Displays animated logo for 6 seconds.
• HomeScreen: Dashboard with navigation cards.
• SettingsScreen: Manages device connection and SD status.
• RetrieveDataScreen: Handles data retrieval and CSV operations.
• ReportsScreen: Displays historical charts.
• LiveChartsScreen: Shows real-time charts.

6.2 Data Models
• ConnectionStateModel: Manages BLE connection and data parsing.
• DataStateModel: Handles CSV storage and persistence.

6.3 Firmware Functionality
• BLE communication with UART service.
• SD card logging and status reporting.
• Command processing (e.g., GET_DATA, SYNC).

7 Testing Instructions
7.1 Test Environment Setup
• Run Flutter app on Android with Bluetooth enabled.
• Load code.py on device with SD card.
• Monitor logs via Logcat.

7.2 Test Cases
1. App Launch: Verify splash screen and transition.
2. Bluetooth Connection: Scan and connect to device.
3. SD Card Status: Check status display.
4. Time Synchronization: Verify auto and manual sync.
5. Data Retrieval: Test incremental and full retrieval.
6. Live Charts: Confirm real-time visualization.
7. Reports: Validate historical charts.
8. CSV Management: Test clean, clear, and share operations.
